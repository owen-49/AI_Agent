## Slide 2 | 问题与动机

**标题：** 核心矛盾 — LLM 的创造力 vs. 数学的严谨性

**三个具体痛点：**
- 高等代数命题依赖专家经验，人力成本高
- LLM 直接出题存在幻觉：矩阵特征值含无理数、矩阵不对称、正交性不满足
- 传统"LLM 构造 + 验证"模式中，Gram-Schmidt 正交化必然引入无理数

**我们的核心洞察：**
> 不要让 LLM 做它不擅长的事（构造数学对象）。
> 让系统程序化构造保证正确的矩阵，让 LLM 只负责教学包装。

**关键约束：**
- 所有特征值必须为整数（适合手算）
- 矩阵必须实对称且元素均为整数
- 构造过程不能使用 Gram-Schmidt（引入无理数的祸根）

---

> **演讲要点（45秒）**
> 高等代数出题有两个核心难点：一是依赖专家，二是如果让 LLM 直接出题，它会胡说八道——常见的问题包括特征值算出来是根号下某个数，或者矩阵根本不对称。
>
> 我们找到的关键问题是：传统做法让 LLM 用 Gram-Schmidt 正交化来构造正交矩阵，但正交化过程必然引入分数甚至无理数。所以我们做了一个根本性的架构决定：不让 LLM 构造数学对象，而是用纯程序化方法生成保证正确的矩阵。

## Slide 4 | 核心创新：MatrixGenerator — 毕达哥拉斯三元组构造法

**标题：** 程序化矩阵生成 — 绕过 Gram-Schmidt，保证整数特征值

**问题：** Gram-Schmidt 正交化涉及开方运算，必然引入无理数。

**我们的解：** 毕达哥拉斯三元组参数化直接构造 2×2 整数特征值对称块

**数学原理：**

对于 2×2 对称矩阵：
```
A = [[a, b], [b, c]]
```

参数化方案（m, n 为整数，m > n）：
```
a - c = ±(m² - n²)
b = m·n
√Δ  = m² + n²     ← 判别式开方恰为整数！
```

此时特征值 `λ = ((a+c) ± (m²+n²)) / 2` 天然为整数。

**从 2×2 到任意尺寸：**

| 尺寸 | 构造方式 |
|------|----------|
| 2×2 | 直接毕达哥拉斯参数化 |
| 3×3 | 2×2 块 ⊕ 1×1 对角块 → 70% 概率随机排列相似变换 |
| 4×4 | 2×2 块 ⊕ 2×2 块 → 70% 概率随机排列相似变换 |

**排列相似变换：** `P·A·Pᵀ` 其中 P 为随机排列矩阵（不改变特征值，但打散块对角结构，使题目"看起来"不像分块矩阵）

---

> **演讲要点（50秒）**
> 这是我们项目最核心的技术创新。传统做法是让 LLM 用 Gram-Schmidt 正交化来构造正交矩阵，但正交化过程涉及归一化——除以向量的模长——必然引入无理数。
>
> 我们换了一个完全不同的思路。从数论出发：对于 2×2 的对称矩阵，它的特征值判别式在特定参数化下恰为完全平方数。这个参数化基于毕达哥拉斯三元组——m²-n²、2mn、m²+n²——其中 m²+n² 天然是一个整数的平方。
>
> 然后我们通过分块对角组合和随机排列相似变换，将 2×2 的构造能力扩展到 3×3 和 4×4。排列相似变换特别巧妙——它不改变特征值，但打散了矩阵的结构，让学生看不出它原来是个分块对角矩阵。
>
> 这意味着我们完全绕过了 Gram-Schmidt 的精度问题，从数学上保证了整数特征值。

## Slide 5 | ReAct 状态机详解

**标题：** ReAct 闭环 — Reason → Act → Observe → Reflect

**状态转移图：**

```
                ┌──────────┐
                │   START   │
                └─────┬─────┘
                      │
          ┌───────────▼───────────┐
          │ REASONING + ACTING    │
          │ LLM 分析矩阵性质      │
          │ 生成题目 + 验证脚本   │
          └───────────┬───────────┘
                      │
          ┌───────────▼───────────┐
          │ OBSERVING             │
          │ 沙箱执行验证脚本      │
          │ 检查特征值整数性      │
          └─────┬──────────┬──────┘
                │          │
           ✅ 通过      ❌ 失败
                │          │
          ┌─────▼──┐  ┌───▼──────────┐
          │  DONE   │  │ REFLECTION   │
          └─────────┘  │ 错误根因分析 │
                        │ 策略调整     │
                        │ 重新生成 JSON│
                        └───┬──────────┘
                            │
                   (最多 3 轮，回到 REASONING)
                            │
                      ┌─────▼──────┐
                      │  FALLBACK   │
                      │ (最后轮次)  │
                      │ 程序化验证  │
                      │ 已通过→接受 │
                      └─────────────┘
```

**四种 Observation 类型：**
| 类型 | 触发条件 | Reflection 注入内容 |
|------|----------|---------------------|
| JSON 解析失败 | 输出不含合法 JSON | 格式要求提示 |
| 脚本为空 | sympy_script 字段缺失 | 要求补充验证脚本 |
| 脚本执行错误 | SymPy/sandbox 报错 | 完整错误信息 |
| 特征值非整数 | 脚本跑通但特征值含小数 | 期望特征值列表 |

**Fallback 机制：** 最后一轮即使 LLM 脚本有瑕疵，只要程序化验证已通过（矩阵本身正确），系统接受输出。

---

> **演讲要点（45秒）**
> 这套 ReAct 状态机是我们质量保障的核心控制回路。每一轮迭代包含四个阶段：Reasoning——LLM 写出数学分析；Acting——输出完整的 JSON 包含题目、解析、验证脚本；Observing——系统将验证脚本投入物理沙箱执行，并检查特征值整数性；如果失败，进入 Reflecting——将具体错误信息反馈给 LLM，要求它分析根因并重新生成。
>
> 我们区分了四种失败类型，每种注入的反馈信息都不同，引导 LLM 有针对性地修正而不是盲目重试。
>
> 一个精巧的设计是 Fallback 机制：即使 LLM 生成的验证脚本有瑕疵，只要程序化验证已经确认矩阵正确，最后一轮系统会接受结果。这避免了因 LLM 的格式细节问题而丢失一道数学上正确的题目。

## Slide 6 | 双重物理沙箱验证

**标题：** MathEngine — 真正的进程隔离执行环境

**设计原则：** 不能信任 LLM 生成的任何代码，必须物理执行验证。

**沙箱架构：**

```
┌─────────────────────────────────────────┐
│         主进程 (Agent)                   │
│   ┌─────────────────────────────────┐   │
│   │  multiprocessing.spawn          │   │
│   │  (非 fork — 兼容 macOS/Windows) │   │
│   └──────────────┬──────────────────┘   │
│                  │                       │
│   ┌──────────────▼──────────────────┐   │
│   │      子进程 (沙箱)               │   │
│   │                                 │   │
│   │  safe_globals 白名单：           │   │
│   │  ├─ sympy, np (仅两个模块)      │   │
│   │  ├─ Matrix, symbols, diag, eye  │   │
│   │  ├─ det, trace, rank, rref      │   │
│   │  ├─ eigenvals, eigenvects       │   │
│   │  ├─ GramSchmidt, diagonalize    │   │
│   │  └─ simplify, solve, factor...  │   │
│   │                                 │   │
│   │  禁止：文件 I/O, 网络, 系统调用  │   │
│   │  超时：10 秒强制 kill           │   │
│   └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**双重验证流程：**
1. **第一重（程序化验证）：** Agent 自己生成验证脚本，确认 MatrixGenerator 产物正确 → 失败则直接终止
2. **第二重（LLM 脚本验证）：** 在沙箱中执行 LLM 生成的验证脚本 → 失败则触发 Reflection

**验证检查点：**
- 矩阵对称性：`assert A == A.T`
- 特征值完整性：总重数 = 矩阵阶数
- 特征值匹配：计算值 vs. 预期值
- 所有矩阵元素为整数
- 所有特征值为 SymPy Integer 类型

---

> **演讲要点（35秒）**
> 沙箱安全是我们非常重视的一个环节。LLM 生成的代码是不可信的，所以我们在一个独立的子进程中执行它，只暴露数学库的白名单接口，禁止任何文件 I/O 或系统调用。使用 spawn 模式而非 fork，保证在 macOS 和 Windows 上都能正确隔离。
>
> 双重验证意味着：系统自己先验一遍矩阵对不对，然后 LLM 的脚本再验一遍。两道关卡都通过才入库。10 秒超时强制终止，防止脚本死循环。

## Slide 7 | 批量评估流水线

**标题：** 大规模异步生成 — 面向数据集构建

**新增模块：** `async_agent.py` + `batch_eval.py`

**架构升级：**

| 特性 | 同步版 (main.py) | 异步版 (async_agent.py) |
|------|-----------------|------------------------|
| LLM 调用 | `invoke`（阻塞） | `ainvoke`（非阻塞） |
| 并发支持 | 无 | asyncio.Semaphore 控制 |
| 适用场景 | Web 交互/单题生成 | 批量数据集构建 |

**batch_eval.py CLI 能力：**

```
# 生成 100 道随机题，5 并发
python batch_eval.py --count 100 --concurrency 5

# 生成 1000 道题，8 并发，支持断点续传
python batch_eval.py --count 1000 --concurrency 8 --resume

# 自定义输出路径 + 固定随机种子
python batch_eval.py --count 500 --output dataset/bank.jsonl --seed 42
```

**断点续传机制：** 从已有 JSONL 中读取 `_meta.index`，跳过已完成的任务。

**输出格式（JSONL）：** 每行一条记录，包含完整题目数据 + `_meta` 元信息（index, topic, difficulty, status）。

---

> **演讲要点（30秒）**

> 批量流水线支持命令行参数配置、断点续传、随机种子固定，输出为标准 JSONL 格式，方便后续的数据集分析和模型训练。实际测试中，5 并发生成 100 道题大约需要 3-5 分钟。

## Slide 9 | Prompt 工程设计

**标题：** 三层 Prompt 体系 — 精确引导 LLM 行为

**Prompt 架构：**

```
┌────────────────────────────────────┐
│  SYSTEM_PROMPT (系统级)            │
│  ReAct 协议定义 + 角色设定         │
│  每次响应必须包含 REASONING+ACTING │
└────────────────────────────────────┘
              ↓
┌────────────────────────────────────┐
│  PROBLEM_PROMPT_TEMPLATE (任务级)   │
│  注入：预计算矩阵 + 特征值          │
│       构造协议说明                  │
│       题目类型 + 难度              │
│       输出 JSON Schema             │
│       五条严禁事项                  │
└────────────────────────────────────┘
              ↓
┌────────────────────────────────────┐
│  REFLECTION_PROMPT (纠错级)        │
│  注入：Observation 错误信息         │
│  要求：根因分析 → 策略调整 → 重输   │
└────────────────────────────────────┘
```

**Prompt 中的关键约束：**
- 禁止修改/替换系统提供的矩阵 A
- 禁止 Gram-Schmidt 构造正交矩阵（引入无理数）
- 禁止非正交可逆矩阵 P 代替正交矩阵 Q
- sympy_script 禁止用反斜杠换行（用括号自然断行）
- 必须输出 `reasoning` 字段（透明化 LLM 思维链）

**4 种随机构造协议（随机选取，增加题目形式多样性）：**
| 协议 | 策略 |
|------|------|
| A | 正交对角化逆构造 + Gram-Schmidt |
| B | 二次型配方法 + 正交变换 |
| C | 特征向量反推 + 谱分解 |
| D | 初等变换与秩 |

---

> **演讲要点（30秒）**
> Prompt 设计分为三层。系统层定义 ReAct 协议和角色；任务层注入预计算矩阵、特征值、以及五条严禁事项；纠错层在验证失败时注入具体错误信息。
>
> 关键的工程决策是：我们在 Prompt 中明确告诉 LLM"这是给你的矩阵，不要改它"，同时禁止它使用 Gram-Schmidt 正交化——因为正是正交化导致了无理数问题。四种构造协议随机选取，给 LLM 提供了不同的教学视角，增加了题目多样性。

## Slide 10 | 基准测试与质量度量

**标题：** 可量化的质量保障体系

**benchmark.py 测试框架：**
- 固定知识点 + 难度，多次运行
- 统计成功率、平均耗时
- 输出 JSON 格式的详细评测报告

**典型测试结果（5 轮 × 二次型标准形 × 难度 4）：**

```
========================================
📊 评测报告 (Benchmark Report)
========================================
总体成功率 (Pass Rate): 100.0%
平均生成耗时: 8.35 秒/题
========================================
```

**成功率的关键保障因素：**
1. 预计算矩阵 → 数学正确性 100% 保证
2. Fallback 机制 → 不会因 LLM 脚本小瑕疵而丢弃正确结果
3. 3 轮 ReAct → 给予 LLM 充分的自我纠正机会
4. 随机排列相似变换 → 矩阵结构多样化
